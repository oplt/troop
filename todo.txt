  SECTION 2: NEW FEATURE RECOMMENDATIONS
  (beyond the product_idea.txt vision)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

--- A. AGENT SIMULATION / SANDBOX ---
[ ] Before activating an agent on a live project, let the user
    run it against a synthetic task set and see how it behaves.
    Catches bad prompts, wrong tool assumptions, budget overruns.

--- B. NATURAL LANGUAGE PROJECT SETUP ---
[ ] "Create a project to build a REST API for payments" →
    manager agent proposes goals, milestones, task list, team.
    User reviews and approves before execution starts.
    Lowers onboarding friction drastically.

--- C. AGENT PERFORMANCE SCORING ---
[ ] Per-agent metrics over time: acceptance rate, avg cost,
    avg latency, review pass rate, escalation frequency.
    Surface underperforming agents and suggest improvements.

--- D. MULTI-REPO / PORTFOLIO DASHBOARD ---
[ ] One orchestrator supervising many repos/projects.
    Cross-project dependency detection
    ("Task A in repo 1 is blocked by Task B in repo 2").

--- E. WORKFLOW TEMPLATES ---
[ ] Pre-built workflow templates:
    - "Bug fix workflow": triage → reproduce → fix → review → PR
    - "Feature workflow": spec → design → code → test → review
    - "Incident response": alert → triage → RCA → patch → post-mortem
    Users can fork and customize templates.

--- F. SKILL PACK MARKETPLACE ---
[ ] Decouple agent identity from reusable skill modules.
    Skills: GitHub Triage, Bug RCA, PR Review, Spec Writer,
    Test Writer, Security Audit, Incident Analyst.
    Agents = identity + selected skills.
    Community-shareable skill packs with version pinning.

--- G. TIME-AWARE AGENT SCHEDULING ---
[ ] Agents can be scheduled on cron patterns.
    "Every Monday at 9am: triage new GitHub issues."
    "After each merged PR: update project knowledge base."

--- H. COST BUDGET ENFORCEMENT ---
[ ] Hard and soft budget caps per project / per agent / per run.
    Soft cap → warn user and pause.
    Hard cap → stop execution + notify.
    Monthly budget dashboards with burn rate projection.

--- I. SECURITY-AWARE AGENT ROLES ---
[ ] Security reviewer agent that runs on every PR:
    - OWASP top 10 scan
    - secret/credential leak detection
    - dependency vulnerability check
    Surface findings as task comments and block merge if critical.

--- J. STRUCTURED OUTPUT SCHEMAS ---
[ ] Each agent can declare an output schema (JSON schema, checklist,
    patch proposal, ADR, issue reply).
    After each run, validate output against schema.
    Reject and re-run if output is malformed.
    Makes agent outputs machine-readable and composable.

--- K. CONTEXT COMPRESSION / TOKEN OPTIMIZER ---
[ ] Before sending long task context to an agent, run a
    compression step: summarize older messages, remove redundant
    information, keep only relevant artifacts.
    Track token savings per project.

--- L. INCIDENT / ALERT INTEGRATION ---
[ ] PagerDuty / OpsGenie / Slack webhook → create incident task.
    Manager agent triages and routes to on-call agents.
    Closes loop by posting resolution back to alert channel.

--- M. PULL REQUEST ASSISTANT MODE ---
[ ] User opens a PR in GitHub → reviewer agent analyses it,
    posts inline comments, requests changes or approves.
    Separate from full task orchestration; lightweight PR review
    mode that works with no project context needed.

--- N. "EXPLAIN THIS RUN" BUTTON ---
[ ] After any task run, generate a plain-English explanation:
    which agent was chosen and why, what tools were called,
    what the agent concluded, what was approved/rejected.
    Aimed at non-technical stakeholders and audit readiness.

--- O. AGENT CONFLICT RESOLUTION LOG ---
[ ] When two agents produce contradictory outputs on the same task,
    log the conflict explicitly and require human or manager agent
    resolution before moving forward.
    Prevents silent inconsistencies in output artifacts.

--- P. CLI / API FOR PROGRAMMATIC CONTROL ---
[ ] REST + WebSocket API for external tools to create tasks,
    trigger runs, and stream results.
    CLI: `troop task create --repo owner/repo --issue 42`
    Enables integration into CI/CD pipelines and IDEs.
